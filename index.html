Executive Summary

The Problem & Proposed Solution

⚠️

The Problem

Vibe needs a configurable gatekeeping system for expenses, invoices, leases, and future operational entities — with per-organization rules, multi-role reviewers, and a durable audit trail. Hard-coding approval logic per entity type is brittle and duplicates policy everywhere.

✅

The Solution

An entity-agnostic Approval Engine that evaluates normalized entity snapshots against immutable, versioned approval rules, records decisions as append-only votes, and returns one of three deterministic outcomes:

✓ Approved ⏳ Pending ✕ Rejected
💡

Key Design Choice

Separate domain entities from approval policy. Expenses, invoices, and leases keep their own domain tables. The engine only needs a stable entity reference, an org, and a normalized set of facts for rule matching — extensibility without engine rewrites.

Scope

Goals & Non-Goals

Goals ✓

  • 🏗️Support expenses, invoices, and leases now
  • 🔌Add future entity types via adapters — no engine rewrites
  • ⚙️Per-org rules using users, roles, vote counts & conditions
  • 🎯Deterministic decisions with clear missing-approval lists
  • 📦Single and bulk approvals/rejections through the same write path
  • 📜Durable audit trail for votes, rule changes & recalculations
  • 🔒Multi-tenant isolation at data, API & auth layers

Non-Goals ✗

  • 🚫General-purpose BPMN / workflow platform
  • 🚫Arbitrary customer code inside rules
  • 🚫Replace existing expense, invoice, or lease lifecycle logic
  • 🚫Notification preferences & escalations (v1)
  • 🚫Real-time collaborative rule editing
Design Principles

Six Guiding Principles

🧩

Entity-Agnostic Core

Logic depends on entity_type, entity_id, org_id, and normalized facts — never domain tables.

🔒

Immutable Rule Versions

In-flight cases bind to the exact rule versions evaluated at creation time. Changes create a new version.

📜

Append-Only Audit

Votes and rule changes are never silently updated or deleted. Corrections are new events.

🎯

Deterministic Decisions

Same case + rule snapshot + vote set → always the same decision. Idempotent by design.

✅

Safe Default

If no rule applies, the entity is immediately approved — no slots to satisfy.

🚫

Sticky Rejection

Once a valid reject vote is cast, that approval case generation is rejected permanently.

Case Lifecycle

Approval Case State Machine

Case generations: if an entity changes materially (e.g. invoice $9k → $90k), the current case is superseded and a new generation is evaluated. Old votes remain audited.

Decision Engine

How Decisions Are Computed

Computation Steps

1

Load the active approval case

2

If any valid reject vote exists → return rejected (sticky)

3

For each matched rule version + slot, count distinct valid approval votes

4

If every slot's count ≥ required count → return approved

5

Otherwise return pending + structured missing list

Key Semantics

🚫

Rejection is sticky for the case generation

✅

Approval requires every applicable rule to be satisfied

⚡

Zero matching rules → immediate approved

👤

User may count once per slot (no double-dipping)

🔁

Decision engine is idempotent

📸

Eligibility checked at vote time and snapshotted

🏢

Role changes after voting don't invalidate historical votes

Missing Approval Response (machine-readable)

{ "status": "pending", "missing": [ { "rule_name": "High value expense requires finance and legal", "slot_key": "finance_review", "required_count": 2, "approved_count": 1, "missing_count": 1, "eligible_selector": { "role_any": [ "finance" ] } } ] }

API Design

REST Endpoints

POST /approval/evaluate

Creates or refreshes an approval case for the exact entity generation. Returns decision + missing approvals.

GET /approval/status

Returns the latest non-superseded case, decision, matched rules, missing approvals, and rejection details.

POST /approval/votes

Single or bulk vote casting. Each item gets its own transaction — per-item results allow partial success. Idempotency keys required.

Rule Management

POST/orgs/{org_id}/approval-rules
PUT/orgs/{org_id}/approval-rules/{id}
POST…/{id}/activate
POST…/{id}/disable
GET…/{id}/versions

Idempotency

Every write requires an idempotency key. Retries with the same key return the prior result — no duplicate votes or cases.

"idempotency_key": "bulk-job-2026-05-21-row-17"
Auditing & Multi-Tenancy

Durability & Isolation

🔒

Immutable Rule Versions

Changes create a new approval_rule_versions row. Old cases keep pointing to old versions. Auditors can prove exactly which rule was in force.

📜

Append-Only Votes

Votes are inserted, never overwritten. Corrections are new events. Rejection is sticky — no silent reversals.

⚡

Transactional Audit Events

Each state-changing write emits an audit_events row in the same transaction. If the audit write fails, the operation rolls back.

Multi-Tenancy Controls

  • ✓Every table carries org_id
  • ✓Row-level security where supported
  • ✓Rule authoring permissions ≠ voting permissions
  • ✓Service-to-service evaluation auth
  • ✓Eligibility resolved only against the case org
  • ✓Rule validation rejects cross-org selectors

Outbox Pattern

State changes and outbox events are written in the same database transaction. A separate publisher delivers them downstream — preventing the failure mode where an approval changes state but the domain service never hears about it.

BEGIN; INSERT INTO approval_cases…; INSERT INTO outbox_events…; COMMIT;

Tradeoffs

Key Design Decisions

❓

Why not hard-code per entity?

Fast initially, but brittle. Every new entity type would duplicate role logic, audit behavior, bulk semantics, and the "what is missing?" response. A central engine keeps these consistent across all current and future entity types.

🔄

Why not a full workflow engine (BPMN)?

Heavier than needed. This problem is about policy matching, quorum-style voting, deterministic status, and auditability. A focused engine is easier to reason about and operate. Events can still integrate external workflow systems later.

📸

Why version rules? Why store entity snapshots?

Versioning creates more rows but is the difference between "we think this was approved correctly" and "we can prove exactly which rule was in force." Snapshots make historical decisions explainable even if the source entity changes later.

Build a Dedicated
Approval Engine
